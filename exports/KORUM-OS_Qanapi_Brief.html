The Full Scoring Lifecycle
Phase 1
Accountability Pass
Automated
After all 5 council phases complete, the Accountability Engine extracts every testable claim and cross-references it across all providers. Claims confirmed by 2+ providers score +40. Legacy crypto without PQC wrappers triggers an automatic −20 penalty. Final score is the average across all extracted claims.
Phase 2
Interrogation
User-Triggered
User selects a provider card and initiates adversarial cross-examination. GPT-4o attacks; Claude defends. Concessions deduct points (−5 to −15). Strong rebuttals with evidence add +3. Scores animate live on each provider card in real-time.
Phase 3
Source Verification
User-Triggered
User highlights any specific claim and sends it to Perplexity for live fact-checking. Verified claims add +5. Partially accurate: −3. Inaccurate: −10. Citations from NIST, MITRE ATT&CK, CVE, and RFC are embedded with every verdict.
Phase 4
Export & Attestation
On Export
Final truth scores are embedded in all exported reports (PDF, DOCX, Excel). Every number in a KORUM-OS report has been tested. The score tells you not just what the AI thinks — but how well it held up under attack, and whether external sources confirmed it.

Why FALCON Matters ML-DSA signatures are approximately 2,420 bytes — larger than a standard network packet. On constrained-bandwidth links (satellites, remote sensors, legacy gateways with sub-1KB packet limits), this causes packet fragmentation and creates a denial-of-service risk. FALCON signatures are approximately 666 bytes — they fit inside existing packet limits without fragmentation. For constrained environments, FALCON is the only quantum-resistant signature that works.

Security & Data Architecture FAQ
Where does the data go?
Queries are sent to cloud AI providers via their official APIs over TLS 1.2+. No data is stored by providers — all 5 have enterprise policies prohibiting training on API inputs. KORUM-OS stores session data locally (SQLite). The Local LLM fallback runs 100% air-gapped.
How are sessions secured?
TLS-encrypted HTTPS for all API calls and user sessions. Security headers enforced (HSTS, X-Frame-Options, CSP). Signed cookies with configurable expiry. Rate limiting at 30 req/min on sensitive endpoints. Input sanitization with 50K char limit. AES-256 session encryption available via Qanapi SDK integration.
Can it run air-gapped?
Yes. KORUM-OS includes a Local LLM provider via LM Studio that runs entirely on-premises. In full air-gap mode, all 5 council seats can be filled by local models — no internet required. Maps directly to SCIF, classified, and regulated environments.
What gets audited?
Every event: council queries, provider responses, interrogations, verifications, logins, failed logins, exports. Includes timestamp, user ID, event type, provider, model, query snippet. Provides full chain-of-custody for intelligence products. Token usage and cost tracked per model.
What does SIGN REPORT do?
Currently staged as an integration hook for Qanapi: FN-DSA (FALCON) digital signature on exported reports. Once connected to Qanapi's Armory, every intelligence product gets a quantum-resistant cryptographic signature providing tamper-evident provenance.
Does it process PII?
No. KORUM-OS processes queries and generates analysis — it does not ingest, store, or process personally identifiable information. This dramatically simplifies GDPR, CCPA, and HIPAA compliance.
Provider failover?
Each provider has an automatic fallback chain: Primary → Mistral Cloud → Local LLM. The council continues with remaining providers. Health is monitored in real-time with color-coded status indicators.
Hallucination prevention?
5-phase sequential pipeline where each AI corrects predecessors. Automated claim extraction + cross-provider verification. Divergence Layer surfaces disagreement. User-triggered interrogation and verification. Confidence defaults to "moderate-to-high." Red Team attacks conclusions before delivery.
