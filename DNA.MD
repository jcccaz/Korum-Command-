# 🧬 KORUM OS — DNA, ENGINE & OUTPUT STANDARD (FINAL)

## 🎯 CORE MISSION

Korum OS is a decision intelligence system that produces engineered intelligence output.

Korum does NOT aim for perfect answers.

Korum converges on the best defensible decision using available information.

## ⚔️ PRIMARY STANDARD (NON-NEGOTIABLE)

If a user would not act on the output with confidence, it does not ship.

## 🧠 OPERATING PHILOSOPHY

Uncertainty is expected
Outputs must still produce a clear decision
Every output must be:
challenged (interrogation)
tested (red team when needed)
structured for execution

## ⚙️ CORE ENGINE (IMMUTABLE)

The following must NEVER change across domains:

Falcon Protocol (orchestration engine)
V1 (parallel answers)
V2 (Architect → Critic → Assembler)
Interrogation + Red Team
Scoring system
Provenance (traceability)
Decision Readiness Gate
Report structure + layout rules

## 🧩 EXECUTION FLOW

🎯 Prompt / Mission Input
⭐ Star Key → refine prompt
🔴 Falcon View → redact sensitive data
👻 Ghost View → user control before execution
🧪 Red Team (optional / required for high-stakes)
🦅 V2 Engine:
Architect → structure
Critic → challenge
Assembler → finalize
⚔️ Decision Arena (4 cards)
roles interrogate each other
scoring adjusts dynamically
📊 Visualization + Docking
📄 Final Report Output

## ⚔️ DECISION READINESS SYSTEM

Every output MUST include:

### 1. Decision Call (Required)

One clear recommendation
No hedging language

### 2. Confidence Band

80–90% → Recommended
70–79% → Conditional (must state what increases confidence)
<70% → FAIL → re-run system

### 3. Why It Wins

3–5 bullets
tied to:
data
interrogation
validated claims

### 4. Risks (Red Team)

Top 3 failure points
must appear INLINE (not buried)

### 5. Action Plan

Format:
① Action (AGENT · Role)
② Action (AGENT · Role)

Must answer:

“What happens Monday morning?”

### 6. Provenance (Required)

Each key claim must show:

origin (agent / data / doc)
challenge path
status:
✓ Verified
⚑ Flagged
◎ Conditional

### 7. Confidence + Status

Score (0–100)
visible status stamp

## 🚫 AUTO-FAIL CONDITIONS

Reject output if ANY exist:

No clear decision
“It depends” language
Walls of text
No action steps
No confidence score
No provenance
Generic formatting
Decorative visuals (no meaning)
Charts not near source data

## 🔁 SYSTEM LOOP (MANDATORY)

IF Decision Score < 80:
→ re-run Critic
→ re-run Assembler
→ strengthen output

ELSE:
→ allow export

## 🧠 REPORT DESIGN RULES (ANTI-GENERIC ENFORCEMENT)

CORE RULE

Content determines layout. NOT templates.

### 🚫 BANNED UI / OUTPUT

Decorative side rectangles
“AI-style” boxes everywhere
Charts placed at bottom of report
Generic tables replacing narrative
Repeated content

### 📊 CHART PROXIMITY RULE (MANDATORY)

A chart MUST appear beside or immediately after its source data.

If not:
→ move it
→ or remove it

### 🧩 LAYOUT STRUCTURE

Strategic Summary
mixed case
first sentence = primary finding
donut/chart floated RIGHT
text wraps LEFT
followed by 3-stat strip

Council Contributors
no large tables
slim row:
Name · Role · Status

Node Sections
prose only
no random boxes/tables
inline attribution

Pull Quotes (MANDATORY USE)
green = verified
red = flagged
blue = strategic

Placed INLINE at moment of relevance

Risk Matrix
high risk → pull quote
medium → small table
do not treat all risks equally

Actions
no tables
numbered execution steps

Composite Truth Score
anchored at bottom
large, visible
closing statement

## 🎨 COLOR SEMANTICS (FIXED)

Blue → structure
Green → verified
Red → risk / flagged
Amber → conditional
